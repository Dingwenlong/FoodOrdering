## 2. 问题单详情

### FO-001 读取项目并整理 README

- 问题背景：项目多端并行开发，启动方式、接口约定、环境切换说明分散。
- 解决思路：
  - 先梳理目录与模块职责，再统一写到一个入口文档。
  - 文档中明确“默认 Mock”与“切真实接口”的最短路径。
- 实现方式：
  - 在主目录新增并持续更新 `README.md`。
  - 补充技术栈、结构、启动步骤、接口清单、环境变量说明。
- 结果：新成员可按 README 独立拉起 backend/admin/miniprogram，并理解当前能力边界。

### FO-002 JDK 17 升级 JDK 25

- 问题背景：需要升级到新 Java 版本，同时保证 Spring Boot 2.7 兼容构建。
- 解决思路：
  - 运行时版本设为 25，编译 release 保持兼容（17），避免直接引入不兼容字节码问题。
- 实现方式：
  - 修改 `backend/pom.xml`：
    - `java.version=25`
    - `maven.compiler.release=17`
- 结果：在当前框架版本下可稳定编译运行，同时满足 JDK 25 环境诉求。

### FO-003 升级 Maven 支持当前 Java

- 问题背景：团队环境 Maven 版本不一致，影响可复现构建。
- 解决思路：
  - 引入 Maven Wrapper，锁定可用 Maven 版本。
- 实现方式：
  - 新增 `backend/mvnw`、`backend/mvnw.cmd`、`backend/.mvn/wrapper/*`。
- 结果：通过 `./mvnw` 或 `.\mvnw.cmd` 保持构建一致性。

### FO-004 管理端真实读接口补齐

- 问题背景：admin 页面原先主要依赖 Mock，真实接口覆盖不足。
- 解决思路：
  - 以管理端页面需要的数据为目标，先补查询接口与 DTO 映射。
- 实现方式：
  - 新增/扩展：
    - `backend/src/main/java/com/foodordering/controller/admin/AdminController.java`
    - `backend/src/main/java/com/foodordering/service/admin/AdminService.java`
    - `backend/src/main/java/com/foodordering/dto/admin/AdminDtos.java`
  - 补齐 notices/users/menu/orders/comments/feedback/support 等读取能力。
- 结果：admin 在关闭 Mock 后可读取真实数据。

### FO-005 管理端写操作接口补齐

- 问题背景：公告、分类、菜品、状态流转等操作无法落到后端。
- 解决思路：
  - 以“最小可用写链路”补齐 CRUD 与状态变更，并增加基础校验。
- 实现方式：
  - 补齐写接口：
    - notices: POST/PUT/DELETE
    - categories: POST/PUT/DELETE
    - dishes: POST/PUT/DELETE
    - users/feedback/support: PATCH 状态
    - orders: POST 状态更新
  - 对应 mapper/entity/schema/data 同步落地。
- 结果：admin 写操作可直连真实后端，不再只停留在 mock。

### FO-006 三端 API 前缀与响应结构统一

- 问题背景：管理端、小程序、后端响应格式不一致，切换 Mock/真实成本高。
- 解决思路：
  - 统一前缀到 `/api/v1`，统一响应 envelope：`{code,message,data,timestamp}`。
- 实现方式：
  - 后端新增：
    - `common/api/ApiResponse.java`
    - `config/ApiResponseAdvice.java`
    - `config/GlobalExceptionHandler.java`
  - admin/miniprogram 请求层新增统一解包逻辑。
  - admin 前缀统一为 `/api/v1/admin`。
- 结果：三端调用模型一致，环境切换只需改开关/基址。

### FO-007 管理端写操作接入页面

- 问题背景：接口虽有，但页面按钮未真正落库。
- 解决思路：
  - 优先接入高频运营页，逐页替换 mock 写逻辑。
- 实现方式：
  - 接入页面：
    - `NoticesPage.vue`
    - `MenuPage.vue`
    - `UsersPage.vue`
    - `FeedbackPage.vue`
    - `SupportPage.vue`
  - 同步扩展 admin `api.ts` 与 mock API。
- 结果：上述页面写操作已贯通，支持真实更新与错误反馈。

### FO-008 管理端历史 TS 报错修复

- 问题背景：`npm run check` 被历史类型错误阻塞。
- 解决思路：
  - 按最新 store/组件签名修正调用方。
- 实现方式：
  - `LoginPage.vue`：改为 `authStore.login({ username, password })`。
  - `OrdersPage.vue`：`StatusPill` 改传 `label/variant`，新增状态映射。
- 结果：`npm run check` 与 `npm run build` 均通过。

### FO-009 补充真实鉴权机制

- 问题背景：原鉴权依赖内存 tokenStore，服务重启即失效且无统一拦截。
- 解决思路：
  - 改为 JWT 签发/校验 + 管理端统一拦截 + 密码哈希校验。
- 实现方式：
  - 后端新增：
    - `auth/AdminJwtTokenService.java`
    - `auth/AdminAuthInterceptor.java`
    - `config/AdminAuthWebMvcConfig.java`
    - `config/SecurityBeansConfig.java`
  - 登录改为签发 JWT；管理接口（除 login）统一 Bearer 校验。
  - 密码升级为 BCrypt，兼容历史明文并自动升级。
  - admin 前端补 401 处理与路由守卫兜底登出。
- 结果：
  - 未带 token 访问管理接口返回 401；
  - 带有效 token 可访问；
  - 鉴权链路具备可持续扩展基础。

### FO-010 小程序端页面与功能补齐

- 问题背景：小程序原流程偏演示态，真实接口/页面交互不完整。
- 解决思路：
  - 先补后端用户侧真实接口，再补小程序请求层契约，最后完善四个核心页面闭环。
- 实现方式：
  - 后端新增用户侧接口：
    - `POST /api/v1/session/bind-table`
    - `GET /api/v1/menu`
    - `POST /api/v1/orders`
    - `GET /api/v1/orders/{id}`
    - `POST /api/v1/pay/wechat/prepay`
    - `POST /api/v1/pay/wechat/confirm`
  - 小程序请求层重构：
    - `miniprogram/utils/request.ts`
    - 支持 `MP_USE_MOCK` 与 `MP_API_BASE_URL`
    - 统一 envelope 解包 + mock/真实同契约
  - 页面补齐：
    - `pages/scan`: 扫码参数解析、手动绑桌、错误态
    - `pages/menu`: 搜索、售罄态、会话校验、下拉刷新
    - `pages/cart`: 清空、备注、金额区、真实下单
    - `pages/orderDetail`: 支付闭环、复制订单号、刷新状态
- 结果：小程序已具备从绑桌到支付确认的完整可运行链路（Mock/真实均可切换）。
