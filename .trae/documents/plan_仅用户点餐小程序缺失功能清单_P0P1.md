# 目标
输出一版“仅针对用户点餐小程序”的缺失功能清单，按 P0/P1 分级，只保留真正应放在用户端的能力，不混入运营后台职责。

# 约束与口径
1. 产品终端口径：仅有“用户点餐小程序 + 运营后台”，不存在店长端小程序。  
2. 归属原则：凡是运营治理、配置管理、审核审批、统计后台类能力，默认归运营后台，不纳入用户端缺失清单。  
3. 用户端原则：只纳入直接影响“点餐-支付-履约-售后-复购”用户闭环的能力。  
4. 现状基线：以仓库当前代码为准，不假设未落地需求文档。

# 实施步骤
1. 盘点用户小程序现有能力（基线）  
   - 扫描小程序路由、页面、接口调用与状态流转。  
   - 形成“已具备能力清单”（如扫码绑桌、菜单、下单、支付、催单、评价）。

2. 建立“应在用户端”的候选能力池  
   - 从点餐业务完整旅程拆解阶段：到店前/点餐中/支付中/履约中/完成后。  
   - 每阶段列出用户应可感知且应可自助完成的关键能力候选（不含后台治理项）。

3. 做差异对比并筛选真正缺失项  
   - 将候选能力与现有基线一一映射。  
   - 仅保留“用户端缺失且后台可承接或可快速补齐”的条目。  
   - 对每个条目标注缺失类型：页面缺失、接口缺失、流程断点、状态反馈不足。

4. 按 P0/P1 分级  
   - P0：影响下单成功率、支付成功率、履约确定性、用户可恢复性的问题。  
   - P1：影响体验与复购但不阻断核心交易链路的问题。  
   - 每项给出分级理由（业务影响 + 发生频率 + 实施复杂度）。

5. 输出最终清单（面向执行）  
   - 每项输出固定字段：能力名称、用户场景、当前缺口、建议方案、所需接口、优先级。  
   - 同步给出“明确不纳入用户端”的能力边界清单（防止后续范围漂移）。

# 交付物结构
1. 总览：用户端应做/不应做边界。  
2. P0 清单：必须优先补齐项（含改造建议）。  
3. P1 清单：次优先补齐项（含改造建议）。  
4. 不纳入项：明确归运营后台的能力说明。  
5. 下一步：可直接进入实现的最小迭代包（按批次）。

# 验收标准
1. 清单中每项都能映射到具体用户场景，不出现后台治理项。  
2. 每项都有明确缺口描述与可执行改造建议。  
3. P0/P1 分级可解释、互斥且无遗漏关键交易环节。
