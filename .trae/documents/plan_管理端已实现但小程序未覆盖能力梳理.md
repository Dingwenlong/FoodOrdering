# 目标
梳理“后台（尤其管理端）已经实现，但小程序端尚未接入或未提供对应能力”的功能清单，并给出证据与建议优先级。

# 范围定义
1. 后台范围：`backend` 中管理端与用户端已存在的业务接口、服务能力、数据模型。
2. 小程序范围：`miniprogram` 中已落地页面、已调用接口、已实现交互流程。
3. 判定标准：后台存在稳定能力（接口与服务可用），但小程序没有页面入口、没有接口调用、或调用后未形成完整业务闭环。

# 实施步骤
1. 建立后台能力清单  
   - 扫描管理端 Controller 与 Service，按模块归类（如公告、评论、留言、工单、订单、菜单、统计、权限等）。  
   - 提取每项能力的接口路径、请求方式、关键入参与返回结构。

2. 建立小程序能力清单  
   - 扫描小程序页面、路由、API 请求封装与调用点。  
   - 提取已上线能力的业务入口、接口依赖和关键状态流转。

3. 做一一映射与差异识别  
   - 以“能力粒度”建立对照表：后台能力 ↔ 小程序能力。  
   - 标注差异类型：完全缺失、部分缺失、流程断点、字段不匹配。  
   - 对每个差异给出代码证据（文件与行号）。

4. 评估可落地性与业务价值  
   - 基于小程序餐饮点餐主链路，判断每个缺失项的优先级（P0/P1/P2）。  
   - 识别可直接复用的后台接口与需要新增适配层的项。  
   - 形成最小改造路径（先接入、后增强）。

5. 输出交付  
   - 输出“后台已实现但小程序未覆盖”清单（模块、现状、证据、影响、建议）。  
   - 给出推荐的接入顺序与最小实施包，便于下一步直接进入实现。

# 交付物格式
1. 总览结论（是否存在、规模、影响主链路程度）。  
2. 差异清单表（能力、后台证据、小程序证据、缺失类型、优先级）。  
3. 下一步执行建议（按优先级拆分为可实施任务）。

# 假设与约束
1. 以当前代码仓库现状为准，不依赖历史分支和未提交设计文档。  
2. 不改动业务代码，仅做调研与规划输出。  
3. 若出现同名能力多实现，优先以已被现网流程调用的实现作为基线。
