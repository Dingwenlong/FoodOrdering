# 后台客服管理支持在线回复消息 - 实施计划

## 一、需求概述

为现有的客服工单系统增加在线消息回复功能，使管理员能够在后台与客户进行实时对话交流。

### 当前状态
- 已存在 `support_tickets` 工单表，记录用户咨询主题和状态
- 管理后台已有工单列表和详情展示页面
- **缺失**：消息历史记录表和消息发送/接收功能

### 目标功能
1. 客户可以发送消息给客服
2. 管理员可以在后台查看消息历史并回复
3. 支持消息列表分页加载
4. 工单详情页面集成聊天界面

---

## 二、技术方案

### 2.1 架构设计
采用 **RESTful API + 轮询** 方案（简化实现，无需WebSocket）：
- 消息发送：POST API
- 消息接收：GET API（支持分页）
- 新消息检测：前端轮询或手动刷新

### 2.2 数据库设计

#### 新增表：`support_ticket_messages`
```sql
CREATE TABLE IF NOT EXISTS support_ticket_messages (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    ticket_id BIGINT NOT NULL COMMENT '关联工单ID',
    sender_type VARCHAR(20) NOT NULL COMMENT '发送者类型：USER/ADMIN',
    sender_id VARCHAR(100) COMMENT '发送者ID（用户昵称或管理员ID）',
    sender_name VARCHAR(100) COMMENT '发送者显示名称',
    content TEXT NOT NULL COMMENT '消息内容',
    is_read TINYINT DEFAULT 0 COMMENT '是否已读：1是，0否',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_messages_ticket_id (ticket_id),
    INDEX idx_messages_create_time (create_time),
    FOREIGN KEY (ticket_id) REFERENCES support_tickets(id) ON DELETE CASCADE
) COMMENT='客服工单消息表';
```

#### 修改表：`support_tickets`
- 现有字段保持不变
- `last_message_at` 字段将由消息表触发更新

## 六、风险评估

| 风险 | 影响 | 缓解措施 |
|-----|------|---------|
| 消息表数据量大 | 性能下降 | 添加索引，考虑分页 |
| 并发消息写入 | 数据不一致 | 使用数据库事务 |
| XSS攻击 | 安全漏洞 | 消息内容转义处理 |

## 七、后续优化建议

1. **WebSocket 实时通信**：替换轮询方案，实现真正的实时消息推送
2. **消息已读回执**：显示对方是否已读消息
3. **图片/文件消息**：支持多媒体消息类型
4. **消息搜索**：支持按关键词搜索历史消息
5. **客服分配**：支持多客服分配和转接
